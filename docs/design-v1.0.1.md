# Design Document: Version 1.0.1 — Refactoring & Data Model Unification & Versioning

---

## 1. Цели версии и мотивация

### Проблема

Архитектура приложения излишне усложнена для текущих задач:

- Существуют модули-прослойки (`Trader-common`), которые не несут самостоятельной ценности, а лишь дублируют код.
- Модели данных размазаны по модулям (`trader`, `test-trader`, `common`), что затрудняет внесение изменений и повышает
  риск ошибок.

### Цель

Упростить архитектуру ядра (`core`), устранить дублирование и унифицировать модели данных.

### Критерии успеха

- [ ] Модули `Trader` и `Test-Trader` объединены в один модуль (`Core`).
- [ ] Модуль `Trader-common` удалён.
- [ ] Количество DTO и дублирующихся доменных сущностей сокращено за счёт унификации.
- [ ] Функциональность версии 1.0.0 полностью сохранена (полная обратная совместимость).

---

## 2. Функциональные требования (изменения для пользователя)

Поскольку версия 1.0.1 является исключительно рефакторингом, **для конечного пользователя не меняется ничего**.

### Сценарии использования

Все 4 команды консольного интерфейса работают идентично версии 1.0.0:

1. **Поиск паттернов** — без изменений.
2. **Тестовая торговля** — без изменений.
3. **Реальная торговля** — без изменений.
4. **Выход** — без изменений.

### Что не входит в эту версию

- **Новый функционал не добавляется.** Любые задачи, не связанные с рефакторингом, переносятся в версию 1.0.2 и далее.

## 3. Технический проект (план изменений в кодовой базе)

### 3.1. Рефакторинг модулей ядра (`core`)

#### Текущая ситуация

В проекте три подмодуля внутри `core/`:

- `trader`
- `test-trader`
- `trading-common`

#### План изменений

**1. Объединение `Trader` и `Test-Trader`**

**Выбранный подход:** один модуль с разными реализациями одного интерфейса и динамическим планировщиком.

- Определяется интерфейс `TradingExecutor` с методом `start()`.
- Создаются две реализации:
    - `BybitRealTradingExecutor` (реальная торговля)
    - `BybitTestTradingExecutor` (тестовая торговля)
- Класс `BybitTradingScheduler` **становится единым** (удаляется дубликат из бывшего `test-trader`).
- Внутри `BybitTradingScheduler` появляется коллекция `List<String>` или `List<Enum.TYPE>`.
- Уже существующий метод, который запускает торгового исполнителя, **расширяется**:
    - Он проходит по списку/набору требуемых исполнителей.
    - Для каждого разрешённого исполнителя создает нужную реализацию `TradingExecutor` и вызывает метод
      `executor.start()`.
    - Это позволяет запускать одновременно и тестовую, и реальную торговлю, если они обе разрешены и присутствуют в
      конфигурации,
      а так же в дальнейшем и реализации различных бирж/брокеров.

**Источник конфигурации:** список исполнителей задаётся через вызов метода `addExecutor(final String executor)`
или `addExecutor(final Enum.TYPE executor)`.

**2. Удаление `Trader-common`**

- Полностью удалить модуль `trading-common`.

### 3.2. Унификация моделей данных

#### Задача

Модели данных (DTO, доменные сущности) могут быть разбросаны по модулям и дублироваться. Необходимо:

1. Составить **полный список** дублирующихся/похожих классов.
2. Принять решение по каждому классу.

#### Список дублирующихся классов

| Класс     | Модуль 1 | Модуль 2      |  Модуль 3  | Различия                                                                                                                                                                                                                                                                                                   | Решение                                                                                                                                                                                                                                                                                                                                                                                                                                     |
|:----------|:---------|:--------------|:----------:|:-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|:--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `Order`   | `Common` | `Trader`      |   `---`    | Класс `Order` из модуля `Trader`, используется для связи с сущностью в базе данных                                                                                                                                                                                                                         | Класс `Order` из модуля `Common` переименовать в класс `OrderDTO` и оставить там только необходимые поля. Класс `Order` из модуля `Trader` оставить без изменений.                                                                                                                                                                                                                                                                          |
| `Pattern` | `Trader` | `Test-Trader` | `Analyzer` | Класс `Pattern` имеет различные реализация в разных модулях. Среди некоторыых различий нет, среди других различи в количестве полей. По факту все эти разновидности делятся на сущность БД и `DTO`. Просто из-за особенностей архитекутры приложения пришлось дублировать меняя лишь наименования классов. | Дублирующие классы в модулях `Trader` и `Test-Trader` удалятся при слияние данных двух модулей. Останется только проблема с тем, что данная сущность нужна модулю `Analyzer` и `Core`. На данный момент было принято решение создать новый модуль репозиторный, конкрентно для данной сущности, модули `Core` и `Analyzer` будут подключать репозиторный модуль как зависимость. В дальнейшенм данное решение возможно будет пересмотренно. |

#### Правила унификации

- **Классы полностью идентичны** → оставляем **один** в модуле `common`, удаляем дубликаты.
- **Классы имеют небольшие различия** → создаём унифицированную версию в `common`, добавляя опциональные поля или
  используя наследование (базовый класс + специфичные наследники, если это оправдано).
- **Классы только кажутся похожими** (разная ответственность) → не трогаем, фиксируем здесь причину.

## 4. План миграции и последовательность действий

### Шаг 1: Анализ кодовой базы (~1 час)

- Составить полную карту дублирующихся классов.
- Определить все зависимости от модулей `Trader` и `Test-Trader`.
- Заполнить таблицу в разделе 3.2.

### Шаг 2: Унификация моделей (~2-3 часа)

- Создать общие классы в модуле `common`.
- Обновить импорты во всех модулях, которые зависели от дубликатов.
- **Чек-поинт:** Убедиться, что проект компилируется без ошибок.

### Шаг 3: Слияние `Trader` и `Test-Trader` (~3-4 часа)

- Перенести код из `test-trader` в `trader`.
- Реализовать выбранный механизм разделения логики (интерфейс / профиль).
- Удалить зависимости от `test-trader` в других модулях (`Console-UI`).
- **Чек-поинт:** Убедиться, что проект компилируется.

### Шаг 4: Удаление `Trader-common` (~1 час)

- Перенести оставшийся уникальный код в целевые модули (`common`, `trader`).
- Удалить модуль `Trader-common`.
- Обновить корневой `pom.xml` (убрать модуль из списка).

### Шаг 5: Финальное тестирование (~1.5 часа)

- Запустить все существующие unit-тесты.
- Провести ручное тестирование по чек-листу (Раздел 7).
- Убедиться, что приложение собирается командой `mvn clean package`.

---

## 5. Нефункциональные требования

- **Обратная совместимость:**
    - Скрипты сборки (`mvn clean package`) должны работать без изменений.
    - Запуск приложения и консольные команды не должны измениться.
- **Схема БД:**
    - Изменения в коде не должны требовать правок в существующих таблицах (`test_orders`, `pattern` и др.).
- **Логирование:**
    - Уровень и формат логирования должны остаться прежними.
