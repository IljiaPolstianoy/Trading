# Design Document: Version 1.0.1 — Refactoring & Data Model Unification & Versioning

---

## 1. Цели версии и мотивация

### Проблема

На данный момент существует конфликт библиотек логирования, а так же отсутствует правильное версионирование проекта:

- Библиотека `Bybit` подтягивает свою библиотеку для логирования, `logback`, а в проекте используется `log4j2`, что
  вызывает конфликт
- На данный момент отсутствует нормальное версионирование проекта. В `pom` файлах до сих пор идет версия `1.0-SNAPSHOT`

### Цель

Исправить конфликт библиотек логирования, а так же привести весь проект к одному стандарту версионирования.

### Критерии успеха

- Проект компилируется и запускается без конфликта библиотек логирования.
- Во всем проекте используется единый продакшен стандарт версионирования.

---

## 2. Функциональные требования (изменения для пользователя)

Поскольку версия 1.0.2 является исключительно рефакторингом, **для конечного пользователя не меняется ничего**.

### Сценарии использования

Все 4 команды консольного интерфейса работают идентично версии 1.0.1:

1. **Поиск паттернов** — без изменений.
2. **Тестовая торговля** — без изменений.
3. **Реальная торговля** — без изменений.
4. **Выход** — без изменений.

### Что не входит в эту версию

- **Новый функционал не добавляется.** Любые задачи, не связанные с рефакторингом, переносятся в версию 1.0.2 и далее.

## 3. Технический проект (план изменений в кодовой базе)

### 3.1. Замена логироавния во всех библиотеках

#### Текущая ситуация

В проекте в моудлях `Market-Data-Bybit` и `Exchange-Connector-Bybit` подключается библиотека `Bybit`, которая
подтягивает зависимость
`logback`.

#### План изменений

**1. Исключить зависимость `logback` во всех подключениях библиотеки `Bybit`**

**Выбранный подход:** везде используется фасад `slf4j`, за реализацию отвечает главный модуль `Core`.

- В модулях `Exchange-Connector-Bybit` и `Market-Data-Bybit` при подключение зависимости `Bybit` добавить исключение
  _транзитивной библиотеки_ `logback`
- В модулях `Exchange-Connector-Bybit` и `Market-Data-Bybit` добавить фасад `slf4j` c scope=provided

**Источник конфигурации:** конфигурационный фалй `log4j2.xml` добавляется  _ресурсы_ модуля `Core`

**2. Логирование для тестирования**

- В модули `Exchange-Connector-Bybit` и `Market-Data-Bybit` добавить зависимость `log4j2` с _scope=test_ для тестов.
- В модули `Exchange-Connector-Bybit` и `Market-Data-Bybit` добавить конфигурационные класс `log4j2-test.xml` для
  тестирования.

---

### 3.2. Настройка версионирования

#### Задача

На данный момент используется старый стандарт версионирования приложения.

#### План изменений

Необходимо во всем проекте заменить версионирование на новый продакшен стандарт.

---

### 3.3. Ключевые зависимости

```xml

<dependensies>
    <!--Пример основных зависимостей-->
    <dependency>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-api</artifactId>
    </dependency>

    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
    </dependency>

    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-slf4j2-impl</artifactId>
    </dependency>

    <!--Пример исключения зависимости-->
    <dependency>
        <groupId>io.github.johnnywic</groupId>
        <artifactId>bybit-java-api</artifactId>
        <exclusions>
            <exclusion>
                <groupId>ch.qos.logback</groupId>
                <artifactId>logback-classis</artifactId>
            </exclusion>
        </exclusions>
    </dependency>
</dependensies>
```

## 5. Нефункциональные требования

- **Обратная совместимость:**
    - Скрипты сборки (`mvn clean package`) должны работать без изменений.
    - Запуск приложения и консольные команды не должны измениться.
- **Схема БД:**
    - Изменения в коде не должны требовать правок в существующих таблицах (`test_orders`, `pattern` и др.).

## 7. Оценка задач и чек-лист готовности

### Задачи

- **Задача 1:** Исключения везеде зависимости `logback`
- **Задача 2:** Добавить везде фасад `slf4j`
- **Задача 3:** Добавить релаизацию `log4j2` в модуль`Core`
- **Задача 4:** Удалить лишние файлы `log4j2.xml`, оставить только в главном `Core` модуле
- **Задача 5:** Добавить в все модули зависимости `log4j-core` и `log4j-slf4j2-impl` с _scope=test_ для тестирования
- **Задача 6:** Заменить во всем проекте версионирование на новый продакшен стандарт
- **Задача 7:** Финальное тестирование. Убедиться, что приложение собирается, запускается и отсутствуют конфликты
  зависимостей логирования

### Чек-лист финального тестирования

- Команда `1` (Поиск паттернов) — отрабатывает корректно.
- Команда `2` (Тестовая торговля) — отрабатывает корректно.
- Команда `3` (Реальная торговля) — отрабатывает корректно (только проверка старта, без реальных сделок).
- Команда `4` (Выход) — приложение завершается без ошибок.
- Сборка `mvn clean package` проходит успешно.
- Итоговый `pom.xml` корневого проекта имеет все нобходиме зависимости с контролем версий.
